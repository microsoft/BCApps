# Prod Order Creation Wizard

The Production Order Creation Wizard creates released production orders directly from purchase lines in a subcontracting context. When a vendor is contracted to manufacture an item, this wizard walks the user through selecting (or creating) the Production BOM and Routing, previewing the resulting components and operations, and committing a fully linked production order back to the purchase line. It exists as a separate module because the process is inherently multi-step -- it must detect what manufacturing data already exists, let the user override it, manage BOM/Routing version lifecycle, and wire up the production order to the subcontracting purchase flow.

## How it works

The wizard is triggered by running codeunit `Subc. Create Prod. Ord. Opt.` with a Purchase Line as the record parameter. The `OnRun` trigger calls `CreateProductionOrderWithTemporaryData`, which is the top-level orchestrator that validates the purchase line, determines the scenario, runs the wizard UI, and commits the result.

Scenario detection happens in `DetermineScenarioAndPrepareData`. The code checks the Stockkeeping Unit first (matching on the purchase line's Location Code and Variant Code), falling back to the Item card's `Production BOM No.` and `Routing No.` fields. The combination of BOM and Routing presence maps to one of three `SubcScenarioType` values: `BothAvailable` (BOM and Routing found), `PartiallyAvailable` (only one found), or `NothingAvailable` (neither found). The `Subc. Management Setup` table then maps each scenario to a pair of `Show/Edit Type` values -- one controlling the BOM/Routing selection steps, the other controlling the Components/Routing preview steps. If both are set to `Hide`, the wizard skips user interaction entirely and builds the temporary structure silently.

When the wizard UI runs (the `SubcPurchProvisionWizard` NavigatePage), the user can select a different BOM or Routing, choose versions, optionally create new versions, and edit lines. All editing happens on temporary records held by `SubcTempDataInitializer`. When the user clicks Finish, the orchestrator calls `TransferTemporaryDataToRealTables`, which is decorated with `[CommitBehavior(CommitBehavior::Ignore)]` for transaction safety. This procedure creates or updates real BOM/Routing versions if the user made modifications, creates a Released Production Order, transfers temp components and routing lines to real tables, moves reservation entries from the purchase line to the new prod order line, and updates the purchase line with the production order number, routing info, and work center linkage. Finally, it cleans up any temporary BOM or Routing records that were created but not needed.

## Things to know

- `SubcCreateProdOrdOpt` is 1198 lines with 67 procedures -- it is the largest codeunit in the Subcontracting app and the central orchestrator for the entire wizard flow.
- The `SubcScenarioType` enum (BothAvailable, PartiallyAvailable, NothingAvailable) determines which wizard steps are shown. The `Subc. Management Setup` table has six fields (two per scenario) that map each scenario to a `Show/Edit Type` for BOM/Routing steps and for Components/ProdRouting steps.
- All user edits happen on temporary records managed by `SubcTempDataInitializer`. Nothing is committed to real tables until the user clicks Finish. If the user cancels, the orchestrator raises `Error('')` and the transaction rolls back.
- When a user modifies BOM or Routing lines in the wizard, new versions are created -- the original certified versions are preserved. The `SaveBOMVersionIfRequired` and `SaveRoutingVersionIfRequired` procedures generate new version codes from No. Series and insert certified version records.
- The `Always Save Modified Versions` setting on `Subc. Management Setup` controls cleanup behavior. When it is off and the user chose not to save BOM/Routing back to the source (Item or SKU), newly created versions are deleted during cleanup.
- `SubcTempProdOrdBind` creates "dummy" real Prod. Order Line and Routing Line records so that temporary routing line validation logic (which checks for a parent prod order line) works correctly. These dummies are cleaned up when the wizard page closes.
- Vendor context flows through a `SingleInstanceDictionary` -- `SubcTempDataInitializer.BuildTemporaryStructureFromBOMRouting` stores the vendor's subcontracting location code under the key `SetSubcontractingLocationCodeFromVendor`, and the `SubcTempProdOrderComp` page reads it back when the user changes a component's Subcontracting Type.
