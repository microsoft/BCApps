# Integration

The integration layer dispatches E-Document operations (send, receive, check status, cancel) to external services via pluggable interfaces. Partners implement these interfaces to connect Business Central to specific e-invoicing networks (PEPPOL access points, tax authority APIs, etc.). The framework handles error trapping, status management, telemetry, and HTTP communication logging so that integrations only need to implement the service-specific logic.

## How it works

The `Service Integration` enum (6151) is the extensibility point. Partners add a new enum value and declare which interfaces it implements. The base enum requires `IDocumentSender`, `IDocumentReceiver`, and `IConsentManager`. Additional optional interfaces (`ISentDocumentActions`, `IDocumentResponseHandler`, `IReceivedDocumentMarker`, `IDocumentAction`) are detected at runtime using `is` checks (e.g., `IDocumentSender is IDocumentResponseHandler`). This means a single enum value can implement as many or as few capabilities as the external service supports.

**Send flow**: The `E-Doc. Integration Management` codeunit orchestrates sending. It retrieves the exported blob from logs, wraps it in a `SendContext`, and dispatches to `IDocumentSender.Send` via the `Send Runner` codeunit. The runner is executed with `if not Runner.Run()` to trap runtime errors. After the call, the framework checks whether the sender also implements `IDocumentResponseHandler` -- if so, it marks the document as async (`Pending Response`) and a background job (`E-Document Get Response`, triggered via Job Queue) will poll `GetResponse` until the service confirms receipt. If the sender does not implement `IDocumentResponseHandler`, the send is synchronous and the status goes directly to `Sent` (or the custom status set via `SendContext.Status().SetStatus()`).

**Receive flow**: Receiving is two-phase. First, `IDocumentReceiver.ReceiveDocuments` is called with a `Temp Blob List` -- the implementation adds one blob per available document (metadata or IDs). The framework then iterates the list and calls `IDocumentReceiver.DownloadDocument` for each entry, passing the metadata blob back so the implementation can extract a document ID and fetch the actual content. If the receiver also implements `IReceivedDocumentMarker`, the framework calls `MarkFetched` after a successful download, allowing the service to mark the document as consumed so it is not re-fetched.

**Action framework**: The `Integration Action Type` enum (extensible) implements `IDocumentAction`. Two built-in actions exist: `Sent Document Approval` and `Sent Document Cancellation`. Each is backed by a codeunit that casts the service's `IDocumentSender` to `ISentDocumentActions` and calls `GetApprovalStatus` or `GetCancellationStatus`. The `E-Document Action Runner` executes these with error trapping. The action returns a boolean indicating whether the service status should be updated. Default statuses are pre-set before the call (e.g., Approved/Approval Error for approval), so implementations only need to return `true` to accept the default or override via `ActionContext.Status().SetStatus()`.

**Context objects**: `SendContext`, `ReceiveContext`, and `ActionContext` are public codeunits that encapsulate state for each operation. They all expose `Http()` (returning `HttpMessageState` for request/response) and `Status()` (returning `IntegrationActionStatus` for the target service status). `SendContext` and `ReceiveContext` additionally carry a `TempBlob` for document content. This pattern replaced the earlier V1 approach of passing multiple `var` parameters (HttpRequest, HttpResponse, IsAsync, TempBlob) through every method.

## Things to know

- The `"E-Document Integration"` interface and the `"E-Document Integration"` enum (the V1 service integration) are obsolete since 26.0 and gated behind `#if not CLEAN26`. All new integrations should use the `Service Integration` enum (V2) and the interfaces in `Integration/Interfaces/`.
- Async detection is implicit: if the `IDocumentSender` implementation also implements `IDocumentResponseHandler`, the framework treats the send as async. There is no explicit `IsAsync` flag to set in V2. The V1 pattern of returning `IsAsync` as a `var Boolean` is deprecated.
- Every runner codeunit (`Send Runner`, `Get Response Runner`, `Receive Documents`, `Download Document`, `Mark Fetched`, `E-Document Action Runner`) calls `Commit()` before `if not Runner.Run()`. This is required by the AL error-trapping pattern but means partial state is committed before the external call. If the call fails, the committed state remains.
- `IntegrationActionStatus` carries both a success status and an error status (via `SetErrorStatus`/`GetErrorStatus`). The built-in action implementations pre-set these before calling the partner's interface method (e.g., `Approved` + `Approval Error` for approvals, `Canceled` + `Cancel Error` for cancellations).
- Batch sending is supported: when `SendRunner` calls `IDocumentSender.Send`, the `EDocument` record parameter can contain multiple records (set by filters). The `E-Doc. Recurrent Batch Send` codeunit groups pending-batch documents by document type, exports them as a batch blob, and sends once per type.
- The `E-Document No Integration` codeunit is the null-object implementation -- it implements all interfaces with empty method bodies and serves as the default for the `"No Integration"` enum value.
- HTTP request and response objects set on the context's `Http()` are automatically logged to integration logs by the framework after each operation. Implementations do not need to log communication manually.
