# Processing

The processing engine orchestrates all e-document operations -- export (outbound), import (inbound), document creation, and background job management. It is the largest area of the E-Document Core app, with 120+ objects. The import pipeline is the most complex subsystem, implementing a multi-step state machine (V2) alongside a legacy single-pass path (V1).

## How it works

**Export flow.** `E-Doc. Export` (codeunit 6102) drives outbound processing. When a BC document posts, `E-Document Subscribers` (codeunit 6103) catches the posting event and calls `CreateEDocumentFromPostedDocument`. The export creates an E-Document record, populates header fields from the source RecordRef, runs the `IExportEligibilityEvaluator` interface to decide whether each service should handle the document, calls the `E-Document` format interface (Check, Create, or CreateBatch) to produce the blob, then logs the blob via `E-Document Log`. Field mapping (`E-Doc. Mapping`) translates between BC field names and the format's expected names, applied in both directions. The workflow engine (`E-Document Created Flow`) then dispatches the exported blob to integration services. Batch processing is supported -- services flagged `Use Batch Processing` skip per-document export and instead get a combined set of mapped RecordRefs in `CreateBatch`.

**Import pipeline (V2).** `E-Doc. Import` (codeunit 6140) orchestrates inbound processing. The V2 pipeline, driven by `Import E-Document Process` (codeunit 6104), implements a 4-step state machine with 5 statuses: Unprocessed, Readable, Ready for draft, Draft ready, and Processed. Each step transitions between adjacent statuses: Structure received data, Read into Draft, Prepare draft, and Finish draft. The pipeline can advance forward to any desired status, or roll back by undoing steps in reverse order. `GetEDocumentToDesiredStatus` calculates the delta between current and target status, then runs the necessary steps or undo-steps sequentially. Each step is wrapped in `Codeunit.Run` with error capture -- failures log to the error message framework and set the service status to `Imported Document Processing Error`. The V1 legacy path collapses all steps into a single `V1_ProcessEDocument` call when the service's import process version is 1.0.

**Provider-based resolution during Prepare draft.** `Prepare Purchase E-Doc. Draft` (codeunit 6125) implements `IProcessStructuredData` and runs during the Prepare draft step. It resolves vendors through `IVendorProvider` (VAT ID, GLN, Service Participant, name+address fallback), resolves units of measure through `IUnitOfMeasureProvider`, and matches lines through `IPurchaseLineProvider` (item reference lookup, text-to-account mapping). After basic resolution, it chains three AI codeunits: historical matching, G/L account matching, and deferral matching. Each operates on the remaining unmatched lines. The `E-Doc. Providers` codeunit (6124) is the default implementation of all four provider interfaces.

**AI/Copilot integration.** The AI subsystem has two independent systems. (1) The `AI/Tools/` codeunits (`E-Doc. Historical Matching`, `E-Doc. GL Account Matching`, `E-Doc. Similar Descriptions`, `E-Doc. Deferral Matching`) implement both `AOAI Function` and `IEDocAISystem` interfaces. They are invoked during Prepare draft to match unresolved purchase lines to historical posted purchase invoice lines using GPT-4.1, with function calling to return structured matches. (2) `OrderMatching/Copilot/E-Doc. PO Copilot Matching` is a separate system for matching imported lines to existing purchase order lines -- it uses a different prompt stored in Azure Key Vault and runs in the V1 order matching flow.

**API surface.** `E-Document Business Events` (codeunit 6185) raises `ExternalBusinessEvent` notifications when E-Document status or service status changes, enabling Power Automate and external integration. API pages (`EDocumentsAPI`, `EDocFileContentAPI`, `EDocumentServicesAPI`, etc.) expose E-Document data and file content via OData/REST.

**Order matching (V1).** When an incoming invoice targets a purchase order, the V1 path (`ProcessExistingOrder` in `E-Doc. Import`) populates `E-Doc. Imported Line` records and links the e-document to the PO via `E-Document Link` on the Purchase Header. The `OrderMatching/` pages let users manually match imported lines to PO lines, creating `E-Doc. Order Match` records. Copilot can propose matches based on description similarity and cost thresholds.

## Things to know

- Two import pipeline versions coexist. The service's `GetImportProcessVersion()` determines whether V1 or V2 runs. V1 calls `V1_ProcessEDocument` which does everything (parse XML, resolve items, create purchase doc) in one pass. V2 uses the 4-step state machine with rollback.
- `E-Document Purchase Line` uses a dual-namespace convention: fields 3-100 hold raw external data (Product Code, Description, Unit Price -- read-only), fields 101-200 hold BC-resolved values prefixed with `[BC]` (Purchase Line Type, Purchase Type No., Unit of Measure, Deferral Code). This separation preserves the original invoice data alongside BC matches.
- The `E-Doc. Record Link` table creates transient links between draft records (E-Document Purchase Header/Line) and BC records (Purchase Header/Line). These links are consumed and deleted when the purchase invoice posts -- at that point they become permanent history in `E-Doc. Purchase Line History` and `E-Doc. Vendor Assign. History`.
- Background jobs use the `EDocument` job queue category. Import jobs run on a recurrent schedule, get-response jobs poll every 5 minutes, and each e-document creation spawns its own one-shot job via `StartEDocumentCreatedFlow`.
- Export eligibility is pluggable via `IExportEligibilityEvaluator`. The default implementation (`Default Export Eligibility`) always returns true. Extensions can implement the interface to suppress export for specific document types or services.
- For Inventory Pick flows, E-Document creation is intentionally deferred. The `AllowCreateEDocument` guard in Subscribers returns false when `InvtPickPutaway` is true during `Sales-Post`. Instead, E-Documents are created in `OnAfterPostWhseActivityCompleted`, after all posting work (including related inbound transfers) has committed.
- The AI matching pipeline is additive and ordered: item reference and text-to-account mapping run first (deterministic), then historical matching (AI), then G/L account matching (AI), then deferral matching (AI). Each step only processes lines that prior steps left unmatched (`[BC] Purchase Type No.` = '').
