# Bug 624741 — Automatic creation of Inspection on Release at posting

## Root Cause Analysis

There are **two distinct issues**:

### Issue 1: Duplicate inspection created on post

**Flow:** User releases PO → inspection created (In Progress) → posting fails (blocked) → PO reverts to Open → user finishes inspection → posts PO again → BC internally re-releases the PO → `HandleOnAfterReleasePurchDoc` fires again → creates a **new** inspection.

**Why:** With the default "Inspection Creation Option" = `"Use existing open inspection if available"` (line 518-526 in `QltyInspectionCreate.Codeunit.al`), the logic finds the existing **finished** inspection and decides `NeedNewInspection := true` because it's no longer Open. This creates an infinite loop: every post attempt creates a new inspection that blocks the post.

### Issue 2: False notification with "Use any existing inspection if available"

**Flow:** With this option, `NeedNewInspection` is `false` when an existing inspection is found (line 527-531). But `HasInspection` is still `true` (line 564-568), so `CreateInspectionWithMultiVariants` returns `true`. The caller at line 384-387 then calls `TrackCreatedInspection(QltyInspectionHeader."No.")`, and `EndBatch` shows the notification.

**Why:** The callers in `QltyReceivingIntegration` don't distinguish between "new inspection created" and "existing inspection found." They track **any** inspection for notification.

## Proposed Changes

| # | File | Change |
|---|------|--------|
| 1 | `QltyReceivingIntegration.Codeunit.al` | In `HandleOnAfterReleasePurchDoc`, detect if the release is triggered during a posting flow and skip inspection creation. Alternatively, check whether a matching finished inspection already exists and skip re-creation. |
| 2 | `QltyInspectionCreate.Codeunit.al` | Expose the `InspectionIsNew` flag via a new getter (e.g., `GetIsNewInspection(): Boolean`) so callers can distinguish new vs reused inspections. |
| 3 | `QltyReceivingIntegration.Codeunit.al` | In `AttemptCreateInspectionWithPurchaseLineAndTracking` (and similar methods), only call `TrackCreatedInspection` when `QltyInspectionCreate.GetIsNewInspection()` returns `true`. |

### Key Risk

Issue 1 fix must not break the legitimate case where a user manually releases a PO a second time and genuinely wants a new inspection. Need to verify if there's a reliable way to distinguish posting-triggered release from manual release.
