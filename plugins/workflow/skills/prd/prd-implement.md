# Task List Management

> **Usage**: Invoke when implementing a PRD. Works through all tasks continuously, using sub-agents for parallel groups.

## Prerequisites

**FIRST**: Read `.github/copilot/project-config.md` for feature toggles:

- **Azure DevOps Integration**: If OFF, skip ADO sync and work item references
- **Memory Bank**: If OFF, skip memory bank updates
- **GitHub Projects**: If OFF, skip GitHub issue updates

Make sure to follow the rules in `.github/copilot/task-workflow.md`.

If the user doesn't pass the name of the PRD, ask first.

## Scope Resolution

Determine the execution scope from the user's argument:

### No argument (run all)

Execute every phase in tasks.md from the first incomplete task to the end. This is the default behavior.

### Phase specified (e.g., `Phase 2`, `2`)

Execute only the tasks within the matching phase:

1. Find the phase header matching the argument (e.g., `### Phase 2:` or phase number `2`)
2. Execute all incomplete tasks (`[ ]` or `[>]`) within that phase
3. **Stop** after the last task in that phase -- do NOT continue to the next phase
4. Report phase completion status to the user

### Sub-task specified (e.g., `2.3`)

Start execution from the specified sub-task:

1. Find the sub-task matching the number (e.g., `2.3`)
2. Begin execution from that sub-task
3. Continue through the remaining tasks in the phase and subsequent phases (same as "run all" but with a starting offset)

### Ambiguity handling

If the argument doesn't clearly match a phase or sub-task number, ask the user to clarify before proceeding.

## Before Starting (per task-workflow.md)

1. **Check project-config.md** - Determine which integrations are enabled
2. **Read tasks.md** - Find the current task, review acceptance criteria and dependencies
3. **Read design.md** - Understand architecture and design decisions
4. **Discover related docs** - Use Glob with `docs/**/*.md` to find completion summaries
5. **Read `.github/copilot/documentation-organization.md`** to understand the flat file documentation rules before creating code
6. **If ADO = ON**: Check Phase header for ADO Feature link and phase status

## Task Implementation

- **Run continuously:** Execute all tasks in order without pausing for user approval. Move immediately from one sub-task (or parallel group) to the next.
- **Parallel groups use sub-agents:** When you encounter a `**Parallel Group**` marker, launch all sub-tasks in that group simultaneously using the Task tool. See **Parallel Group Execution** below.
- **Task status markers** (per task-workflow.md):
  - `[ ]` - Not started
  - `[>]` - In progress (add Started timestamp when marking)
  - `[x]` - Complete (add Completed timestamp and Duration)
- **Timestamp format:** `YYYY-MM-DD HH:MM:SS UTC+-X` (e.g., `2025-10-09 14:30:00 UTC-7`)
- **Completion protocol:**
  1. When you finish a **sub-task**, mark it `[x]` with Completed timestamp and Duration.
  2. If **all** subtasks underneath a parent task are now `[x]`, also mark the **parent task** as completed.

## Parallel Group Execution

When the task list contains `**Parallel Group**` markers (generated by `/prd tasks`), execute those sub-tasks concurrently using sub-agents instead of sequentially.

### How it works

1. **Complete prerequisite tasks first** - Any sequential sub-tasks before a parallel group must finish before the group starts.
2. **Launch sub-agents in parallel** - Use the Task tool to launch one sub-agent per sub-task in the group. Send all Task tool calls in a single message so they run concurrently.
3. **Provide context to each sub-agent** - Each sub-agent prompt must include:
   - The specific sub-task description and acceptance criteria
   - Relevant documentation references from the parent task
   - The design.md location and key design decisions that apply
   - The implementation directory preferences (already confirmed with user)
   - Any shared interfaces, types, or contracts defined by prior tasks that the sub-task depends on
4. **Wait for all sub-agents to complete** - Monitor all launched agents. Do not proceed to the next task until every sub-agent in the group has finished.
5. **Review and reconcile** - After all sub-agents complete:
   - Verify there are no conflicts between the parallel outputs (e.g., conflicting imports, duplicate definitions)
   - Mark all sub-tasks in the group as `[x]` with timestamps
   - Update tasks.md with implementation notes from each sub-agent
6. **Resume sequential flow** - Immediately continue with the next task after the parallel group (typically an integration or wiring task)

### Example flow

Given this task list:

```markdown
- [ ] 2.0 Build API and UI components
  - [ ] 2.1 Define shared interfaces and types
  - **Parallel Group A** (after 2.1 completes):
    - [ ] 2.2 Implement API endpoint with validation
    - [ ] 2.3 Create frontend component with mock data
    - [ ] 2.4 Write database migration script
  - [ ] 2.5 Integration testing (after Parallel Group A completes)
```

The execution order is:

1. Execute 2.1 (implement, mark complete)
2. Launch three sub-agents simultaneously for 2.2, 2.3, and 2.4
3. Wait for all three to complete, review results
4. Execute 2.5 (implement, mark complete)

## Task List Maintenance

1. **Update tasks.md after each sub-task:**
   - Mark task `[x]` with Started, Completed, Duration
   - Add implementation notes
   - Update phase Progress count

2. **Update design.md when architecture changes:**
   - Document design decisions made during implementation
   - Add "Last Updated" timestamp to modified sections

3. **Create/update component documentation:**
   - When a phase creates code in a technical component, ensure `[component]/docs/` files exist and are current
   - Follow the flat file pattern from `.github/copilot/documentation-organization.md` (architecture.md, implementation.md, setup.md, testing.md, CLAUDE.md)

4. **Create Completion Summary** (for significant work):
   - Location: `docs/tasks/TASK-X.Y.Z-TASK-NAME-COMPLETION-SUMMARY.md`
   - Include: What was implemented, design decisions, implementation details

5. **Maintain "Relevant Documentation" section:**
   - Update documentation references as implementation progresses
   - Add any new files created with a one-line description

## Implementation File Location

Before creating implementation files, determine the correct location by:

1. **Check existing project structure** - Use Glob and Grep to infer where files belong (e.g., existing components in `src/components/`, APIs in `src/api/`)
2. **Follow established conventions** - Match the patterns already in the codebase
3. **If no convention exists** - Ask the user once at the start for directory preferences, then apply consistently throughout

## AI Instructions

When working with task lists, follow task-workflow.md:

**BEFORE starting:**

1. Read tasks.md, design.md, and related completion docs
2. Ask for implementation directory preferences before creating files
3. Check which sub-task is next

**DURING work (sequential tasks):**

4. Mark task `[>]` with Started timestamp when beginning
5. Implement the sub-task
6. Immediately proceed to the next task

**DURING work (parallel groups):**

4. Mark all tasks in the group `[>]` with Started timestamp
5. Launch sub-agents in parallel via the Task tool (one per sub-task, all in a single message)
6. Wait for all sub-agents to finish, then review and reconcile outputs
7. Immediately proceed to the next task

**AFTER each task or group:**

8. Mark task(s) `[x]` with Completed timestamp and Duration
9. Update design.md if architecture changed
10. Create completion summary for significant work (in `docs/tasks/`)
11. Update phase Progress count in tasks.md

## Workflow Context

This skill is part of a 3-step workflow:

1. `/prd design` - Generate the design document (PRD)
2. `/prd tasks` - Create task list from PRD
3. `/prd implement` - Execute tasks (this skill)
