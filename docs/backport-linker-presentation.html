Backport Work Item Linker

Automating Parent-Child ADO links for backported PRs

microsoft/BCApps · GitHub Actions · Azure DevOps REST API

Problem

When a bug is fixed on main and backported to a release branch, two separate ADO work items are created. Today, developers must manually link them and fill KB fields.

Tedious & Error-Prone

Open ADO, find both work items, manually add the Parent-Child relation. Wrong parent, wrong link type — no validation.

Often Forgotten

No enforcement. Work items end up orphaned. KB Article and KB Title left blank.

Solution

A GitHub Actions workflow — link-backport-workitems.yml — that automatically detects backport PRs on release branches, creates the Parent-Child link, and fills KB fields.

How It Works

Patterns & Safety

Supported Patterns

Auto backports #NNN / backport of #NNN from CrossBranchPorting

Manual Fixes / Closes / Resolves #NNN from developer PRs

Safety

Idempotent — checks if link exists before creating

Self-link guard — skips if both WI IDs match

Graceful — no AB# or issue? Logs warning, exits cleanly

Read-only GitHub — only writes to ADO

KB Field Updates

The workflow populates KB Article and KB Title on the release work item from the GitHub issue title.

How

Fetches the issue title via GitHub API. Sets both KB Article and KB Title to the same value.

If unavailable, KB fields are skipped — link still created.

Example

Issue title Fix for posting error in Sales Invoice
KB Article Fix for posting error in Sales Invoice
KB Title Fix for posting error in Sales Invoice

Configuration (One-Time)

1. ADO PAT

  • Org: dynamicssmb2
  • Scope: Work Items → Read & Write

2. GitHub Secret

  • Name: ADO_PAT
  • Settings → Secrets → Actions

3. Release Branches

  • Workflow file must exist on each releases/** branch
  • Future branches inherit from main
