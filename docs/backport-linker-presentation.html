Backport Work Item Linker

Automating Parent-Child ADO links for backported PRs

microsoft/BCApps · GitHub Actions · Azure DevOps REST API

Problem

When a bug is fixed on main and backported to a release branch, two separate ADO work items are created.

Today, developers must manually open ADO and create a Parent-Child link between them.

Tedious

Open ADO, find both work items, manually add the relation. For every single backport.

Error-Prone

Wrong parent, wrong link type, or linking to the wrong work item — no validation.

Often Forgotten

No enforcement. Work items end up orphaned with no traceability between main and release fixes.

Concerns & Constraints

Different PR Patterns

Automated backports use backports #NNN.
Manual PRs use Fixes #NNN.
Both must be supported.

AB# Is on PRs, Not Issues

The work item reference (AB#) lives in the PR description, not the issue. We need to follow the chain: issue → timeline → find the PR.

Idempotency

Workflow can trigger multiple times (PR edited, labeled). Must not create duplicate links.

Two Separate Issues

If main and release PRs reference different issues, there's no chain to follow. Requires a shared issue or direct PR reference.

Solution

A GitHub Actions workflow — link-backport-workitems.yml — that automatically detects backport PRs on release branches and creates the Parent-Child link in ADO.

How Resolution Works

The workflow follows a chain to connect the release PR back to the main PR's work item:

Release PR #7734 │ ├── body contains AB#631297 ← release work item (child) └── body contains Fixes #7732 ← shared issue reference │ ▼ Issue #7732 Timeline API (mockingbird preview header) │ ├── cross-referenced by PR #7733 ← found it! └── cross-referenced by PR #7734 ← (skip, that's us) │ ▼ PR #7733 body contains AB#631296 │ ▼ PATCH: 631297.parent = 631296

Supported Patterns

Pattern Source Example
backports #NNN Automated CrossBranchPorting "This pull request backports #7729 to releases/28.x"
backport of #NNN Automated CrossBranchPorting "backport of #7729"
Fixes #NNN Manual Developer PRs "Fixes #7732"
Closes #NNN Manual Developer PRs "Closes #7732"
Resolves #NNN Manual Developer PRs "Resolves #7732"

The backport pattern takes priority. Both PRs must share a common issue or the release PR must directly reference the main PR number.

Safety Guarantees

Idempotent

Checks if the parent link already exists before creating. Safe to re-run on edits, labels, or manual dispatch.

Self-Link Guard

Skips if release and master work item IDs are the same. No accidental self-referencing.

Graceful Degradation

If no AB# or issue reference is found, logs a warning and exits cleanly. No failures, no noise.

Read-Only GitHub

Only reads PR descriptions and issue timelines from GitHub. Write access is limited to ADO work item relations.

When Does It Run?

Event Condition
pull_request_target: opened New PR opened against releases/**
pull_request_target: edited PR description edited (e.g., AB# added later)
pull_request_target: labeled Label applied (e.g., "Linked" from WorkitemValidation)
workflow_dispatch Manual trigger with PR number (for testing)

The job only executes if the PR body contains AB# — otherwise it skips immediately.

PRs targeting main are not affected. This workflow only handles releases/** branches.

Known Limitation

Two Different Issues

If the main PR references Fixes #100 and the release PR references Fixes #200 (different issues), the workflow cannot find the connection.

The workflow needs a shared link between the two PRs:

  • Both PRs reference the same issue
  • Release PR uses backports #<main_PR> (direct reference)

The CrossBranchPorting tool always writes backports #NNN, so this only affects fully manual backport PRs with different issues.

Configuration (One-Time Setup)

1. ADO Personal Access Token

  • Organization: dynamicssmb2
  • Scope: Work Items → Read & Write
  • Create at dev.azure.com → Profile → Personal access tokens

2. GitHub Repository Secret

  • Name: ADO_PAT
  • Value: the PAT from step 1
  • Add at GitHub → Settings → Secrets → Actions

3. Workflow File on Release Branches

The file must exist on each releases/** branch. Future branches cut from main inherit it automatically. For existing branches:

git checkout releases/28.x && git checkout main -- .github/workflows/link-backport-workitems.yml
